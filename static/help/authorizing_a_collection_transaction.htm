Authorizing a Collections Transaction

About Transaction Authorization

Every payment request that has been entered into the system has to be authorized. This feature follows a Maker-Checker concept wherein one user known as the Maker enters the transaction and another user known as the Checker authorizes the transaction. In the Maker-Checker concept it is mandatory for the Maker and Checker to be different users. The Checker can either authorize or reject a transaction at Payment Request level. Unless the checker authorizes the transaction, it is not used as the active record in the system nor is it processed as the valid transaction. By default, the Payment Request pending for authorization having the entry date same as the application date is displayed. In case you are assigned Across Seller rights for a corporation, you can authorize transactions of clients spread across different countries.

Authorization Levels

Every collection request that has been entered into the system has to be authorized following the Authorization Level set in the My Product Master Entry. Authorization of a single instrument transactions is always at ‘Instrument Level’ but for Deposit, it depends on the Authorization Level field set in the Collection MyProduct master. If you have selected ‘Batch Level’, then checker will authorize the entire Deposit amount and credit the client’s account with one credit entry. If you have selected ‘Instrument Level’, then checker will authorize every instrument within a Deposit individually and credit the client’s account with number of credit entries equivalent to the instruments within the Deposit.

•    At the time of authorization, in case the validations are incorrect, the authorizer has the right to Reject or Discard a transaction. The Maker can 'Modify' (edit) a Rejected transaction, 'Update' the changes and immediately make it available for authorization. The modified record does nor require the 'Submit' process. At this stage, event notification will be triggered to notify the authorizer as 'Transaction pending for Auth'.

Super User Authorization

•    Each instruction has to undergo the authorization process as per the authorization matrix that has been set up. Only after the successful completion of the authorization process, an instruction can be sent to the bank for further processing. In situations where in the authorizers that are involved in the authorization process is unavailable, the authorization process may not proceed at all. This may result in a delay in sending that instruction for further processing to the bank. As a solution, GCP CashWeb front end offers the capability to the super user to authorize an instruction irrespective of the authorization matrix definition associated with it. A super user can authorize transaction at batch as well as instrument level

IMP:

-         All transactions initiated in GCP CashWeb have to be authorized and sent to bank for further processing within a time frame set in the My Product master.  At both these stages, the Client Checker  (when the status of the transaction is Pending My Auth) and the Client, (when the status of the transaction is Pending Release) will receive a notification of the pending action.

-         In case a transaction remains unauthorized, system will notify the client before scrapping the transaction based on the ‘number’ of days entered in the ‘Notify User of Scrap before’ field in the MyProduct master.

-         Based on the ‘Scrap if not Auth or Send’ limit set in the MyProduct master by the Maker, all transactions with status- ‘Pending Auth’ and ‘Pending My Auth’, exceeding the limit will be scrapped bt the system and notification of the same will be sent to the Client and the Client Checker (authorizer). This is applicable for all the GCP CashWeb Client initiated transactions.  

Authorizing a Transaction and Releasing it to the Bank:

In the Collection center screen, select the record with status showing Pending My Auth and click the . On successful authorization the screen will display a message confirming the same. The status of the authorized record changes to - Pending Release which indicates that it is ready to be sent to the bank for further processing. Depending on the parameters set up in the My Products master, the transactions are sent manually or automatically.

To reject or delete the transaction, click More and select Reject or Discard. To reject a transaction, it is mandatory to specify a reject reason.

Note:  When the status of the transaction changes to 'Pending Release', client will receive a notification of the same. This event will be fired on authorizing the transaction.  In case a transaction with status 'Pending Release' does not appear in the summary screen, the same may have been scrapped based on the ‘Scrap if not Auth or Send’ limit set in the MyProduct master. Notification of the same will be sent to the Client and the Client Checker (authorizer) of the same. This is applicable for all the GCP CashWeb Client initiated transactions.

Sending Transactions to the Bank

Send’ signifies an action where transactions are sent to the GCP CashWeb Back office application for further processing, after client has initiated and authorized the transactions in GCP CashWeb.

Depending on the parameters set up in the My Products master, the transactions can be sent manually or automatically.

You cannot send the transactions that are on ‘Hold’. You need to first ‘Release’ the transactions from hold and then ‘Send’ to the bank for further processing. In case you are assigned Across Seller rights for a corporation, you can perform this function for clients spread across different countries.

Note: In case Auto Send is selected at MyProduct level for products which do not require an explicit “Send”, the authorized transactions are automatically transmitted to the back office application for further processing. You can use this option to send transactions which are back / current / future dated. Back / current dated transactions are automatically rolled over to the subsequent working day if they cross the cutoff time. Future dated transaction are subsequently warehoused at the bank till their due date.

To ‘Send’ Instructions to Bank for Further Processing of Transaction:

1    In the Payment Summary screen, select the record with status 'Pending Release'. You can only send those transactions for which you are assigned the rights.

2    Click to release the record to the bank for further processing. The status of this transaction now changes to Sent to Bank.

3    In case you wish to discard a transaction at this point, click the . A popup appears where you have to specify the reason for holding or discarding the transaction.

4    Click OK. A message confirming the action appears in the header area of the screen.

Note: All the discarded transactions can be viewed in the Payment Summary screen with the status as Deleted and are available for cloning. Once replicated, it is available in the Payment Request screen for initiating the transaction

5    You also have the option to Hold the transaction.

Note: In case a future-dated transaction on authorization is sent to bank from Client-end, the same will show the status as ‘Warehoused’ instead of Sent to Bank. This ensures the Client is also aware of the pending status of the transaction.

To Hold/ Release a Transaction

GCP CashWeb provides a feature for holding a transaction due to reasons such as insufficient funds for making the collections etc. Only those transactions with status 'Pending Release' can be put on hold. This prevents the transaction from being released to Bank. It is mandatory to release and authorize such transactions kept on hold for further processing. This feature is available only for authorizers with privileges to this screen.

1    In case you wish to hold a transaction at this point, in the Collection center screen, select the transaction and click . A popup appears where you have to specify the reason for holding the transaction.

2    Click OK. A message confirming the action appears in the header area of the screen.

3    To release a document kept on hold, select the transaction and click icon

Note: Every release request has to be authorized. As the authorizer, you need to validate the correctness of the information provided for the hold request and either accept the record or reject it.
