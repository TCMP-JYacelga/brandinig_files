The Maker - Checker Concept

§        Each GCP transaction/instruction that you have entered has to be authorized to make it active/ executable in the system. GCP CashWeb supports the Maker-Checker concept to make sure that only approved transactions or instruction entries are activated in the system. Here, the maker of the transaction is the one who has made the data entry of the transaction. The checker is the one who verifies the authenticity of the records and authorizes the data entered by the maker. The checker essentially has to be someone, other than the maker of the transaction. Unless the checker authorizes the transaction, it is not used as the active record in the system nor is it processed as the valid transaction.

§        A Maker if provided with rights, can Edit, Enable, Disable or Discard a record while a Checker can only View and Authorize a record.

§        In GCP CashWeb Client Console, the record entries that you make are termed as definitions. All definitions that you have entered in the system need to be authorized using the Authorization Menu. These include both, new and modified definitions. Similarly, all requests such as disabling or enabling definitions also need to be authorized. On authorization, the relevant definitions are accordingly enabled, disabled, or rejected.

States Of a Definition:

Every definition in the Client Console is associated with a state that indicates the current action that has been taken on it.

Following are various states that can be associated to a definition depending upon the action performed.  

□         New: Indicates that the definition has been newly added and has not yet been authorized.

□         Draft: An incomplete definition.

□         Enabled: Indicates that the definition has been authorized and saved.

□         Modified: Indicates that an enabled definition that is now being changed, and which has yet to be authorized.

□         Disabled: Indicates a definition that has been disabled and authorized.

□         Disable Request: Indicates that an enabled definition has been disabled, but which has not yet been authorized.

□         Enable Request: Indicates that a disabled definition has been enabled, but for which authorization is pending.

□         Rejected: Indicates that a new definition has been rejected on authorization. It can either be modified or deleted.

□         Mod. Reject: Indicates that a modified definition has been rejected on authorization. It can either be discarded or edited. If discarded, its status will change to that prior to modification.

□         Dis. Req. Reject: Indicates that the disable request for an enabled definition has been rejected during authorization.

□         Ena. Req. Reject: Indicates that the enable request for a disabled definition has been rejected during authorization.

Authorizing a Request

§        Authorization of any request can be described as the process of verifying and approving the request on a definition. This request may be for modifying, enabling, or disabling a definition. In the process, you can either accept or reject the request.

§        A request that is accepted on authorization is an Enabled request, the same can be Disabled. Likewise, you can also enable a disabled request. Disabled, and enabled requests also require authorization. If a request for Disable, or Enable is rejected on authorization, then the request must be discarded. On discarding, the definition is reverted back to its original state.

§        The Definition Entry List tells you at a glance whether a particular definition has been authorized or not. This is indicated in the Status column, which displays the current state of the corresponding definition. Depending on whether the definition has been authorized or not, the Status column takes different values.

  • If Rejected: Enable-Req Reject

In this case, the Enable Request by the Maker has to be discarded so as the definition retains the Disabled state.

A brief explanation on how to authorize a request:

Authorizing a Master

Authorizing a Master:

□         Click Admin >> {Definition} Setup Auth >> The Definition. The “{Definition} Authorization” screen appears. The screen displays all the definitions that are pending for authorization.

□         Select the Accept check box to accept the request.

OR

Select the Reject check box to reject the request. While rejecting a request, it is mandatory to type the reason for rejection. Type the reject reason and click OK.

Note: The reject reason can have a maximum of 128 characters. The system does not support the following special characters: ‘ “ ` ^ \

Tip: To view the maker checker details of the definition, click the History  icon displayed against the definition.

Authorizing a Transaction

Authorizing a Transaction:

□         In the summary screen, filter the list by status-Pending My Auth to retrieve all the transactions that are available to you for authorization.

□         Select the check box under 'Select/ Deselect all Records'; the Authorize icon in the menu bar is enabled.

□         Click the Authorize icon to authorize the transaction. The status of the transaction changes to Pending Release and is now available to be sent to the bank.

You also have the option to Reject or Discard the transaction.
