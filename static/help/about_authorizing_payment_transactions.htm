About Authorizing Payment Transactions

Every payment request that has been entered into the system has to be authorized. This feature follows a Maker-Checker concept wherein one user known as the Maker enters the transaction and another user known as the Checker authorizes the transaction. In the Maker-Checker concept it is mandatory for the Maker and Checker to be different users. The Checker can either authorize or reject a transaction at Payment Request level. Unless the checker authorizes the transaction, it is not used as the active record in the system nor is it processed as the valid transaction. By default, the Payment Request pending for authorization having the entry date same as the application date is displayed. In case you are assigned Across Seller rights for a corporation, you can authorize transactions of clients spread across different countries.

Authorization Levels

Authorization for Quick Pay transactions is always at ‘Instrument Level’ but for Batch Payments, it depends on the Authorization Level field set in the Payment MyProduct master. If you have selected ‘Batch Level’, then checker will authorize the entire Batch Amount and debit the client’s account with one debit entry. If you have selected ‘Instrument Level’, then checker will authorize every instrument within a batch individually and debit the client’s account with number of debit entries equivalent to the instruments within the Batch. Batch Level authorization has an additional step ahead wherein authorization can happen at Batch – Max Instrument Amount level.

E.g.

On ‘Instrument Level’ & ‘Batch Level’ Authorization

-         Consider there are 2 instruments within a file with amount 1000 & 5000 each. AVM setup is as follows:

-         0-1000                             User 1

-         1001-10000               User 2

-         If Authorization Level is at 'Instrument Level', then instrument of amount 1000 will go to User 1 for authorization. Instrument of amount 5000 will go to User 2.

-         If Authorization Level is at 'Batch Level', then total batch amount 6000 will go to User 2 for authorization.

On ‘Batch Level’ and Batch – Max Instrument Amount Authorization

-         Consider the Authorization Type is set to 'Batch level'. The file contains 6 instruments with amount 1000, 2000, 3000, 4000, 5000 & 10000. AVM setup is as follows:

-         0-1000                             User 1

-         1001-10000               User 2

-         10000- ~                       User 3

Case 1: Batch-Max Instrument Amount check box is unchecked.

In this case, system will total the instrument amount to 25000 and put the batch into last slab i.e. 10001- ~ for authorization.

Case 2: Batch-Max Instrument Amount check box is checked.

In this case, system will consider the maximum instrument amount within the batch i.e. 10000 and put the batch into slab 1001-10000 for authorization.

Auto Authorization Limits

Payment transactions available for auto authorization will have an additional check for authorization limits. These authorization limits are set in Payment MyProduct master against a field named Auto Approval Limit. This field will accept only numerical values in seller base currency. The Auto Approval Limit field is added for manually entered as well as for file uploaded transactions. On Submit, system will check the specified approval limit against the Instrument Level, Batch Level or Batch - Max Instrument Amount in seller base currency.

Hence, if the Instrument, Batch or Batch - Max Instrument Amount is less than the Auto Approval Limit (after conversion into seller base currency) and if Approval Required checkbox is unchecked then transaction will be auto authorized.  If Approval Required checkbox is checked then transaction will be authorized as per AVM/SVM setup. If valid AVM/SVM setup is not available then system will raise an exception error.

Note: Ensure that the user sets the Auto Approval Limit as per maximum of any slab in AVM/SVM. In case of mismatch, then AVM/SVM slab will prevail.

Batch Splitting Logic

Batch transactions can be split on the basis of the field Split File Batch By AVM/SVM checked in Payment MyProduct master. This feature will be enabled only when Authorization Level is set to ‘Batch Level’ and Batch-Max Instrument Amount is enabled. If this field is checked then system will split the Batch into multiple PIRs based on AVM/SVM slab depending on the Authorization Type set to ‘My Product’ or ‘Account’. If Authorization Type is set to ‘My Product’ then split the Batch on the basis of batch amount and AVM/SVM slab defined at My Product level. If Authorization Type is set to ‘Account’ then split the Batch on the basis of batch amount and AVM/SVM slab defined at Account level. Below examples will explain the batch split logic clearly.

Note:

-         Batch splitting logic will be applicable only if Approval Required field is unchecked, and Auto Approval Limit (File Upload) field is set to default value i.e. 99999999999.99.

-         Currently, system supports only AVM at account level. While introducing support for SVM at account level, we will allow only users (after validating user & account restriction) as part of SVM expression. Category will not be allowed as part of expression.

-         Effect of changes in AVM/SVM after batch split will not affect the already split batch even if it is in reject repair. Reject repair will be on split batch and file level parameter will apply for split PIR rather than whole file. Hence after repair, no further splitting is to be done.

Example 1: Authorization Type is set to ‘My Product’: Consider the upload file ‘batch_split_tst1.txt’ contains 9 instruments with amount 100, 200, 1000, 1300, 1700, 2000, 2500, 2100 & 3000.

Case 1: Consider MYPROD1 with Authorization Type as ‘MyProduct ‘& Authorization Level as ‘Batch’ & Batch – Max Instrument Amount is enabled.  AVM setup is as follows:

-         0-1000                             User 1

-         1001-2000                  User2

-         2001- ~                           User3

Case 2: Consider MYPROD1 with Authorization Type as ‘MyProduct ‘& Authorization Level as ‘Batch’ & Batch – Max Instrument Amount is enabled.  SVM setup is as follows:

-         0-1000                             User 1

-         0-2000                             User2

-         0- ~                                     User3

In above cases, file will be split into 3 PIRs as follows:

-         PIR1: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 1300 with Total No. of Instruments = 3. In both the cases, PIR 1 will be available to User 1 for authorization since Batch- Max instrument Amount is 1000.

-         PIR2: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 5000 with Total No. of Instruments = 3. In both the cases, PIR 2 will be available to User 2 for authorization since Batch- Max instrument Amount is 2000.

-         PIR3: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 7600 with Total No. of Instruments = 3. In both the cases, PIR 3 will be available to User 3 for authorization since Batch- Max instrument Amount is 3000

 

Example 2: Authorization Type is set to ‘Account’: Consider the upload file ‘batch_split_tst1.txt’ contains 9 instruments with amount 800 & 300 belonging to Account 1, instrument 1000 belonging to Account 2, instruments 1300, 1700 & 2000 belonging to Account 2, instrument 2100 belonging to Account 1, instruments 2500 & 3000 belonging to Account 2. Consider MYPROD1 with Authorization Type as ‘Account‘& Authorization Level as ‘Batch’ & Batch – Max Instrument Amount is enabled.  AVM setup is as follows:

For Account 1

-         0-1000                             User1

-         1001-2000                  User2

-         2001- ~                           User3

For Account 2

-         0-1000                             User4

-         1001-2000                  User5

-         2001- ~                           User6

In above case, file will be split into 5 PIRs as follows:

-         PIR1: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 1100 with Total No. of Instruments = 2. In this case, PIR 1 will be available to User 1 for authorization since Batch- Max instrument Amount is 800.

-         PIR2: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 1000 with Total No. of Instruments = 1. In this case, PIR 2 will be available to User 4 for authorization since Batch- Max instrument Amount is 1000.

-         PIR3: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 5000 with Total No. of Instruments = 3. In this case, PIR 3 will be available to User 5 for authorization since Batch- Max instrument Amount is 2000.

-         PIR4: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 2100 with Total No. of Instruments = 1. In this case, PIR 4 will be available to User 3 for authorization since Batch- Max instrument Amount is 2100.

-         PIR5: Ref No ‘batch_split_tst1.txt’, PIR Total Amount = 5500 with Total No. of Instruments = 2. In this case, PIR 5 will be available to User 6 for authorization since Batch- Max instrument Amount is 3000

Super User Authorization

Each instruction has to undergo the authorization process as per the authorization matrix that has been set up. Only after the successful completion of the authorization process, an instruction can be sent to the bank for further processing. In situations where in the authorizers who are involved in the authorization process are unavailable, the authorization process may not proceed at all. This may result in a delay in sending that instruction for further processing to the bank. As a solution, GCP CashWeb front end offers the capability to the super user to authorize an instruction irrespective of the authorization matrix definition associated with it. A super user can authorize transaction at batch as well as instrument level.
